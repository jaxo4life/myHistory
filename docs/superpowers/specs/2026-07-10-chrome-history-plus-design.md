# Chrome History Plus — 设计规格

- **日期**: 2026-07-10
- **状态**: 设计已确认，待实现计划
- **目标产物**: 一款开源、可审计、纯本地的 Chrome 浏览历史管理扩展，完整复刻 [Browser History Plus](https://browserhistory.net/)（Chrome Web Store ID: `adjifobelhoemalljgnjdlockheiacmc`）的核心体验

---

## 1. 背景与动机

原扩展 Browser History Plus 功能强大但**闭源**。用户出于安全与数据主权顾虑不敢使用——无法审计它是否真的不上传数据。因此本项目目标：**用开源、代码透明的方式完整复刻其体验**，让一切数据与逻辑都在用户掌控之中。

核心价值主张：

- **数据主权**：所有浏览历史只存储在用户本机 IndexedDB，绝不上报任何外部服务器
- **突破 Chrome 自带历史 ~90 天的时效限制**，按用户需要长期保存
- **代码全开源可审计**：采集逻辑集中在单一 service worker，审查一目了然
- **完整复刻**：日历视图、高级搜索、自动分类、标签、分析、批量管理、导入导出

## 2. 技术选型（已确认）

| 维度 | 选型 | 理由 |
|---|---|---|
| 扩展规范 | Manifest V3 | Chrome 当前强制标准 |
| 构建框架 | **WXT** | 现代 MV3 扩展框架，处理 manifest/构建/HMR，体验最好 |
| UI 框架 | **React 18 + TypeScript** | 交互密集（日历、图表、列表）场景成熟，类型安全 |
| 本地数据库 | **IndexedDB + Dexie.js** | 容量大、可索引、异步持久；Dexie 提供声明式 schema 与 `liveQuery` 响应式查询 |
| 设置存储 | `chrome.storage.local` | 小数据、跨 context 读取用户偏好 |
| 样式 | Tailwind CSS（WXT 集成） | 快速实现深/浅双主题与精确像素复刻 |
| 图表 | Recharts（阶段③分析） | React 生态主流图表库 |

## 3. 整体架构

四层结构，职责清晰、可独立理解与测试：

```
┌─────────────────────────────────────────────────────────┐
│  UI 层  (React + WXT: popup / history page)              │
│  日历视图 │ 列表视图 │ 搜索过滤 │ 统计图表 │ 标签管理    │
└───────────────▲─────────────────────────────────────────┘
                │ Dexie liveQuery（响应式订阅）
┌───────────────┴─────────────────────────────────────────┐
│  查询/业务层  (src/lib/)                                  │
│  搜索引擎 │ 过滤器 │ 统计聚合 │ 自动分类 │ 导入导出 │ 备份│
└───────────────▲─────────────────────────────────────────┘
                │ 读写
┌───────────────┴─────────────────────────────────────────┐
│  存储层  IndexedDB (via Dexie.js)                        │
│  visits 表（核心历史） │ settings（用户配置）            │
└───────────────▲─────────────────────────────────────────┘
                │ 写入
┌───────────────┴─────────────────────────────────────────┐
│  采集层  background service worker                       │
│  webNavigation.onCommitted → 解析/过滤 → 写库           │
│  + 启动时 chrome.history 回填                            │
└─────────────────────────────────────────────────────────┘
```

### 3.1 数据采集机制（已确认：webNavigation）

后台 service worker 监听 `chrome.webNavigation.onCommitted`，每次导航提交时记录。

- **选择理由**：最准确，可拿到 `transitionType`（link/typed/redirect/reload…）、真实 URL、frameId，能区分用户主动导航与重定向
- **所需权限**：`webNavigation`、`tabs`（读取标题/favicon）
- **启动回填**：扩展首次安装/更新时，调用一次 `chrome.history.search` 把 Chrome 现存历史（最多 ~90 天）灌入库，让用户装上即有数据
- **被否决的方案**：
  - `tabs.onUpdated`：拿不到 transition 类型，预渲染/快速跳转易漏
  - `history.search` 轮询：本身依赖被限制的数据源（违背突破 90 天的初衷），且有轮询延迟

### 3.2 数据模型（`visits` 表）

**每次访问一条记录，不按 URL 去重**，保留完整时间线——这是日历视图、时间线视图、访问次数统计的前提。"访问次数"由查询时按 URL 聚合得出。

| 字段 | 类型 | 索引 | 说明 |
|---|---|---|---|
| `id` | `number` (auto-increment) | 主键 | 主键 |
| `url` | `string` | — | 完整 URL |
| `domain` | `string` | ✅ | 派生（按域名过滤/分组） |
| `title` | `string` | — | 页面标题 |
| `visitTime` | `number` | ✅ | 访问时间戳（毫秒），日历/排序/范围查询的核心 |
| `dayKey` | `string` | ✅ | 派生为 `YYYY-MM-DD`，日历"哪天有记录"的点亮依据 |
| `transitionType` | `string` | — | link / typed / redirect / reload … |
| `referrerUrl` | `string?` | — | 来源页 |
| `faviconUrl` | `string?` | — | 站点图标 |

设置存储（`chrome.storage.local`）：主题（dark/light）、周起始日（默认周一）、记录标记模式（dot/heatmap）、域名黑名单、分类规则等。

### 3.3 隐私 / 安全边界（默认锁死）

呼应"担心安全"的初衷，以下策略默认开启，不可被静默绕过：

1. **隐身模式不记录**：`tab.incognito === true` 直接跳过
2. **内部/本地页不记录**：`chrome://`、`edge://`、`extension://`、`moz-extension://`、`file://`、`about:`、`view-source:`
3. **域名黑名单**：用户可配排除域名（银行、邮箱等敏感站）
4. **零网络上报**：manifest 不申请多余 host 权限；CSP 限制 `connect-src` 仅允许本地/扩展自身；代码中无任何 `fetch`/XHR 向外部发送历史数据
5. **全开源可审计**：采集逻辑集中在单一 service worker 文件；附 README 隐私声明

## 4. UI 设计

### 4.1 整体布局（三栏，复刻官网截图）

```
┌──────────┬──────────────────────────────┬──────────────┐
│ 左栏 ~25% │      中栏 ~50%               │  右栏 ~25%   │
│          │ 顶部：搜索框 + 时间范围切换   │              │
│  📅 日历  │      (Hour/Day/Week/         │  数据分类列表 │
│          │       4 Weeks/All)           │  Browsing 12  │
│          │                              │  Downloads 3  │
│ 选中日期  │  该日期历史记录列表          │  ...          │
│ 的历史   │  [时间] favicon 标题/URL [×] │               │
│          │  [时间] favicon 标题/URL [×] │  [Clear Data] │
└──────────┴──────────────────────────────┴──────────────┘
```

入口：

- **popup**：工具栏图标点击弹出，提供快速搜索框 + "打开完整历史"链接
- **history page**：独立全屏页（`chrome-extension://<id>/history.html`），承载三栏主界面

### 4.2 左侧日历表（核心，100% 复刻目标）

逐像素规格（基于官网截图分析；实现时以实际渲染验收为准）：

| 维度 | 规格 |
|---|---|
| **网格** | 7 列 × 6 行，单元格约 40×40px，圆角 4px |
| **周表头** | 周一开头 `M T W T F S S`（用户确认），12px 次要灰色 |
| **月份头** | 左右箭头 + 居中 "March 2026"，箭头为强调紫色 |
| **有记录的日子** | 单元格底部居中紫色小圆点（直径 8px） |
| **选中日期** | 单元格填充强调色背景，白字 |
| **今天** | 强调色 1px 描边（outline）；与"选中日期"（填充背景）视觉区分 |
| **上下月灰色日期** | 淡灰色字，无圆点 |
| **深色主题配色** | 背景 `#1E1E1E`、卡片 `#252525`、边框 `#2A2A2A`、主文字白、次要文字 `#8E8E8E`、**强调色 `#6C5CE7` 紫** |
| **字体** | Inter 无衬线；月份 16px / 周头 12px / 日期 14px |
| **标记模式（阶段①交付）** | 默认统一圆点；设置可切热力图色阶（访问越多圆点越深/越大）——用户确认。注意：这是**日历内的渲染选项**，与阶段③「分析仪表盘的热力图」是两回事 |

**技术实现要点**：

- 点亮依据：对 `visits.dayKey` 索引，查询当前可视月份存在哪些 `dayKey`，O(1) 点亮圆点
- 热力模式：同查询额外聚合每个 dayKey 的记录数，映射到色阶
- 选中日期 → `liveQuery` 响应式刷新中栏列表
- 月份切换仅切换可视窗口，不重新拉全量数据
- **主题**：深浅双主题 + 切换（用户确认）；浅色主题为深色版的反色映射

### 4.3 中栏历史列表

- 每条横向卡片（高约 48px，卡片背景、圆角 4px、垂直间距 8px）
- 字段：左侧时间（12px 次要灰）→ favicon → 标题（14px 白）→ URL/域名（12px 强调紫）→ 右侧删除按钮 `×`
- 默认按时间倒序；支持按域名分组（阶段②）

## 6. 分阶段实现路线

每个阶段都是可独立运行、可交付的版本。

### 阶段 ① MVP 核心闭环（首个实现计划范围）

- WXT + React + TS 脚手架
- 采集层：`webNavigation.onCommitted` 监听 + 首次启动 `chrome.history` 回填
- 存储层：Dexie 初始化、`visits` 表与索引
- 左栏日历表：周一起、圆点标记、月份切换、深浅双主题切换
- 中栏：选中日期的历史列表（渲染）
- 隐私过滤（5 条默认策略）
- **验收**：安装扩展 → 正常浏览 → 日历上一眼看到哪天有记录 → 点击某天查看当天列表

### 阶段 ② 搜索与管理

- 搜索框 + 过滤器（域名、日期范围、关键词、访问次数）
- 单条删除 / 多选批量删除
- 右栏分类统计 + Clear Data
- 按域名分组视图

### 阶段 ③ 高级能力

- 11 类自动分类（基于域名的规则映射）
- 自定义标签 CRUD
- 分析仪表盘（Recharts：趋势、Top 站点、热力）
- 导入导出（CSV / JSON / PDF）
- 自动备份（定时 dump 到本地）
- 时间线 / 标签视图

### 阶段 ④ 打磨发布

- 大数据量性能优化（虚拟列表、日历查询缓存）
- 边界情况（超长标题、重复导航合并、回填去重）
- 商店素材 + README 隐私声明
- 发布到 Chrome Web Store（可选）

## 7. 非功能性需求

- **可审计性**：采集逻辑单一文件、零外部网络依赖，README 附隐私自检清单
- **性能**：单月日历点亮查询 < 50ms；万条记录下列表滚动流畅（虚拟化）
- **数据安全**：IndexedDB 数据可一键导出/清除；扩展卸载即清除
- **可维护性**：四层分离，每层单一职责、接口清晰、可独立测试

---

## 8. 开放问题（实现阶段解决）

- 历史记录的**保留策略**：是否提供"自动清理 N 天前数据"选项（默认永久保留）
- **重复导航合并**：短时间内同一 URL 多次 onCommitted（如页面刷新/重定向链）是否去重，去重窗口多长
- **PDF 导出**：客户端生成方案（jsPDF）的样式还原度
- 11 类自动分类的**初始域名规则表**来源（内置种子 + 用户可编辑）
